Duplicated Code: Both `sendNewCommentNotification` and `sendTitleChangedNotification` contain nearly identical `for` loops that iterate over `issue.getWatchers()`, check `user.wantsEmailNotification()`, retrieve the email, check for null, and invoke the email client. Fixing a bug in the notification logic requires updating multiple places.
Refactoring: Extract method: move loop logic into a helper function

Feature Envy: `NotificationMaker` is overly interested in the data of the `User` class. It pulls the user, checks if they want an email, asks for their email address, and checks it for nulls.
Refactoring: Tell, don't ask. The `User` object (or an intermediary service) should encapsulate its own notification preferences. The `NotificationMaker` should just pass the message payload to the `User`, increasing encapsulation.

Message Chains: `issue.getOwner().getName()` relies on `NotificationMaker` understanding the structural relationship that an Issue has an Owner, and an Owner has a Name.
Refactoring: Hide Delegate. Create a method on the `Issue` class called `getOwnerName()`. Reduces tight coupling. If the internal implementation of how an `Issue` stores its owner changes, `NotificationMaker` won't break.

Adding conditional `if (user.wantsSlackNotification())` blocks to every notification method violates the **Open/Closed Principle**. Every time a new notification platform (e.g., SMS, Microsoft Teams) is added, the `NotificationMaker` must be directly modified, which risks breaking existing code and creates a bloated, monolithic class.

Refactoring: Implement the strategy or observer pattern.
- Create a standard `NotificationService` interface with a `sendMessage(User, subject, body)` method.
    
- Create concrete classes implementing this interface: `EmailNotificationService` and `SlackNotificationService`.
    
- Maintain a list of these services. When an event occurs, iterate through the injected services and call `sendMessage()`, letting polymorphism handle the routing.
