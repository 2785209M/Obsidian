An architectural pattern is a high-level plan used to govern how software is organized within a system. The difference between an architectural pattern and a design pattern is scope and abstraction. Design patterns are techniques used to solve problems and keep code maintainable at low-level components, like classes.

(b)
I would consider using Message oriented architecture for this approach, specifically the publisher-subscriber model. With this method you can keep your data decentralized by keeping each lab's data local. It would work by one lab sending a request over a broker as if it were a newspaper advertisement. The broker would send it to every client that is subscribed to it, and each client would process the data and eventually send back a response. This matches how the researcher describes his idea in the specification.

(c)
In order to fit this scenario, this method would need to be asynchronous, and each lab machine would need to install an adapter to maintain their own unique data fields. Step 1: the host sends a request to the broker which distributes the method among subscribers. Step 2: Each subscriber would receive the data and put it through a pre-installed adapter which would translate the request into their own database format. Step 3: The lab machine would process the request, which may take a long time in large datasets. Once the data has finished processing, each subscriber would send back a response asynchronously to the original client using an address that was on the original request. Step 4: The client would collect these responses over time.

# 4. Software Maintenance and Testing
(a)
(i) A mutation is an altered section of code altered to produce a bug for testing purposes
(ii) A mutant is the variant of the core program that has been generated with a mutation
(iii) A survivor is a mutant that was not caught by the testing process
(iv) Effectiveness is the percentage of mutants that were killed during testing
(v) Efficiency is a measure of the resources and time required to achieve a specific level of effectiveness

(b)
These tests do not assert the result to an expected value. They just call the function without checking that is returned the correct thing. This is a serious flaw as you cannot kill mutants without assertions to an expected value.

Besides this, code coverage measures what code was executed, not the correctness of the code.  In this case they are using a "getMessageString" method. It may be that the code coverage only used "happy path" data, and avoided edge cases that would break the program, such as using a datatype other than a string. This specific test code also has three redundant test scenarios using the same string, which is pointless and creates overhead,

(c)
"Negate Conditionals" searchers for conditional statements in your code and flips their logical operators (true becomes false, < becomes >=, etc.).
"Replace Return Value with Default Value" Negates your computed return value and replaces it with meaningless data. Numbers usually return 0, objects come back as undefined, strings come back as "", etc.
In order to kill these mutants we must use the "assertEquals" method in the tests, which compares an expected value to a real value:class GroupChatMessageTest {

```
    @Test
    void testGetMessageStringWithNoParent() {
        GroupChatMessage message = new GroupChatMessage(
                getCurrentUser(), "hi all.", null);
        
        // Kills negated condition (parent != null or speaker != currentUser) and null returns
        assertEquals("You replied: hi all.", message.getMessageString());

    @Test

        GroupChatMessage parent = new GroupChatMessage(
                getCurrentUser(),"hi all.", null);

        // Kills negated condition (parent != null)
        assertEquals("You said: hello.", message.getMessageString());

    @Test
    void testGetMessageStringWithDifferentSpeaker() {
        GroupChatMessage message = new GroupChatMessage(
                new User("Alice"), "hi all.", null);

    }
}
```
